# 04 – Security Considerations

## Overview

Security is a foundational pillar of this architecture, not an afterthought. Every component has been selected and configured with a defence-in-depth approach, applying controls at the edge, application, data, and audit layers. The platform is deployed exclusively in the **AWS Frankfurt (eu-central-1)** region to satisfy EU data residency requirements and align with **GDPR** obligations.

---

## 1. Identity and Access Management (IAM)

### 1.1 Principle of Least Privilege
All Lambda execution roles are scoped to the minimum permissions required for their specific function. No role carries wildcard (`*`) resource permissions. For example, the document upload Lambda is granted `s3:PutObject` on the designated intake bucket only — it cannot read, list, or delete objects.

### 1.2 Amazon Cognito — Role-Based Access Control
Three distinct user groups are defined within the Cognito User Pool, each mapped to a corresponding IAM role:

| User Group | Permitted Actions |
|---|---|
| **Supplier** | Upload documents via pre-signed URL, view own submission status |
| **Internal Reviewer** | Read and annotate submitted documents, update review status |
| **Administrator** | Manage users, access audit logs, configure system parameters |

Cognito issues short-lived **JWT tokens** (ID + Access tokens) upon successful authentication. These tokens are validated by API Gateway on every request, ensuring that no unauthenticated or improperly scoped call reaches the application layer.

### 1.3 Service-to-Service Trust
Inter-service communication (e.g., Lambda → DynamoDB, Lambda → S3) uses IAM resource-based policies rather than hardcoded credentials. No long-term access keys are stored in application code or environment variables.

## 2. Edge and Network Security

### 2.1 AWS WAF (Web Application Firewall)
AWS WAF is attached to the CloudFront distribution and API Gateway, providing protection against:
- **OWASP Top 10** attack vectors (SQL injection, XSS, etc.)
- **Rate-based rules** to mitigate brute-force and DDoS attempts
- **Geo-blocking** rules that can be applied to restrict access to permitted regions

### 2.2 Amazon CloudFront
CloudFront serves as the sole public entry point for the frontend application. Direct access to the S3 static hosting bucket is blocked via **Origin Access Control (OAC)**, ensuring all traffic is inspected by WAF before reaching origin resources.

### 2.3 HTTPS Enforcement
All communication between clients and the platform is encrypted in transit using **TLS 1.2 minimum**, enforced at both the CloudFront and API Gateway layers. HTTP-to-HTTPS redirection is enabled at the CloudFront distribution level.

## 3. Secure Document Upload — Pre-Signed URLs

Rather than routing file uploads through the API Gateway and Lambda (which would increase latency and cost), documents are uploaded directly to S3 using **pre-signed URLs** generated by a dedicated Lambda function.

**Security properties of this approach:**
- The pre-signed URL is **time-limited** (default: 5 minutes), minimising the window for interception or misuse.
- The URL grants access to a **single, specific S3 key** — it cannot be used to access other objects.
- The authenticated supplier's identity is embedded in the S3 object key path (e.g., `uploads/{cognito-user-id}/{timestamp}/{filename}`), enforcing ownership traceability.
- S3 bucket policies deny any `PutObject` request that does not originate from a valid pre-signed URL issued by the trusted Lambda role.

## 4. Data Protection and Encryption

### 4.1 Encryption at Rest
All data stores are encrypted at rest using **AWS KMS (Key Management Service)** with customer-managed keys (CMKs):

| Resource | Encryption Method |
|---|---|
| S3 Document Bucket | SSE-KMS with CMK |
| DynamoDB Metadata Table | KMS-managed encryption |
| CloudWatch Logs | KMS-encrypted log groups |

Key rotation is enabled on all CMKs (annual automatic rotation), and key usage is logged via CloudTrail.

### 4.2 Encryption in Transit
All API calls between AWS services (Lambda → S3, Lambda → DynamoDB, Lambda → SNS) use **AWS private endpoints or TLS-encrypted SDK calls**. No inter-service communication traverses the public internet.

### 4.3 S3 Bucket Hardening
The document storage bucket is configured with:
- **Block All Public Access** enabled
- **Versioning** enabled to protect against accidental deletion or overwrites
- **Object Lock** (WORM mode) available for compliance-sensitive document retention
- **Bucket Policy** explicitly denying `s3:DeleteObject` for all roles except the Administrator

## 5. Monitoring, Logging, and Alerting

### 5.1 AWS CloudTrail
CloudTrail is enabled across all services with a **multi-region trail** writing to a dedicated, tamper-protected S3 bucket. It captures:
- All API calls made to S3, DynamoDB, Lambda, Cognito, and KMS
- User identity, source IP, and timestamps for every action
- Management events and data events for the document bucket

This trail serves as the **primary audit log** for compliance reporting and incident investigation.

### 5.2 Amazon CloudWatch
CloudWatch collects operational metrics and application logs. Key monitors include:
- **Lambda error rates and duration** — alerts trigger if error rate exceeds 1% over a 5-minute window
- **API Gateway 4xx/5xx rates** — elevated client or server error rates indicate misuse or failures
- **DynamoDB throttling events** — surface capacity constraints before they impact users
- **Custom metric filters** on Lambda logs to detect suspicious patterns (e.g., repeated unauthorised access attempts)

### 5.3 SNS Alerting
CloudWatch Alarms are routed through **Amazon SNS** to notify the operations team via email (or integration with external tools like PagerDuty) when thresholds are breached.

## 6. Compliance and Data Governance

### 6.1 GDPR Alignment
Deployment in **eu-central-1 (Frankfurt)** ensures all data remains within the EU, satisfying GDPR data residency requirements. Key compliance measures include:

- **Data minimisation**: only document metadata strictly required for the review process is stored in DynamoDB.
- **Right to erasure**: the Administrator role can trigger S3 object deletion and DynamoDB record removal upon verified request.
- **Consent and access logs**: Cognito login events and CloudTrail API logs provide a verifiable record of who accessed which documents and when.

### 6.2 Separation of Duties
No single user role can perform the full lifecycle of a document submission unilaterally:
- A **Supplier** can upload but cannot approve or delete.
- A **Reviewer** can assess but cannot modify the raw uploaded file.
- An **Administrator** can manage users and audit logs but is not involved in the review workflow.

This separation reduces the risk of insider threats and satisfies common audit control requirements.

---

## 7. Security Design Trade-offs

| Decision | Rationale |
|---|---|
| Pre-signed URLs over API-proxied uploads | Reduces attack surface on the API layer; limits credential exposure window |
| Cognito over custom auth | Managed service reduces implementation risk; battle-tested token validation |
| KMS CMK over SSE-S3 | Provides key access audit trail and supports customer-controlled rotation |
| Single-region deployment | Simplicity and GDPR compliance; multi-region failover deferred to future iteration |
| WAF managed rule groups | Faster deployment vs. custom rules; sufficient for current threat model |
